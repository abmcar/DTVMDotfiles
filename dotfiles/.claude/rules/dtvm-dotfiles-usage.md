# DTVMDotfiles Usage

Treat the checked-in DTVMDotfiles scripts as authoritative. Before answering
any workflow question or editing the system, re-read the relevant script —
this rule is a map of invariants, not a substitute for the source.

## Entry points

- `DTVMDotfiles/release.sh` — push dotfiles SSOT → parent DTVM repo
- `DTVMDotfiles/store.sh` — collect parent DTVM repo → dotfiles SSOT
- `DTVMDotfiles/diff.sh` — detect drift between deployed and dotfiles
- `DTVMDotfiles/worktree-sync.sh` — symlink dotfiles into an existing worktree
- `DTVMDotfiles/worktree-init.sh` — submodule init + dotfiles sync + FetchContent seed for a new worktree (backs the SessionStart hook)
- `DTVMDotfiles/setup_from_dotfiles.sh` — first-time bootstrap of a fresh DTVM workspace
- `DTVMDotfiles/lib/sync_common.sh` — `MIRRORED_ITEMS` definition (what's managed)
- `DTVMDotfiles/dotfiles/exclude.map.sh` — `.git/info/exclude` source

## Invariants

### Manifest-based sync

`release.sh` writes `.claude/.dtvm-manifest.json` in the parent repo
(per-file sha256). Three consequences:

- `release.sh` uses file-level sync (no `rm -rf`).
- `store.sh` only collects files listed in the manifest.
- Unmanaged files (created locally, not from dotfiles) are never touched by
  any of the three scripts. `diff.sh` shows them as "unmanaged".

### What's managed vs. not

Managed: everything in `MIRRORED_ITEMS` — read `lib/sync_common.sh` for the
current list. (As of this writing it covers `.claude/`, `CLAUDE.md`, top-level
utility scripts, and the personal `worktree-bootstrap` skill SSOT.)

NOT managed:
- `CLAUDE.local.md` — per-machine; bootstrapped from
  `dotfiles/CLAUDE.local.md.template` by `setup_from_dotfiles.sh` only when
  missing. `release.sh` / `store.sh` never touch it.
- `.agents/skills/` (other than `worktree-bootstrap/`) — upstream-tracked
  skills. Mirror at `.claude/skills/<name>/SKILL.md` is regenerated by
  `python3 .agents/tooling/generate_skill_mirrors.py` after sync.
- Anything in `.claude/` not originating from dotfiles (diff.sh marks as
  "unmanaged").

### Two CLAUDE.md authority chains

Project-level `~/DTVM/CLAUDE.md` is owned by DTVMDotfiles (this system).
User-level `~/.claude/CLAUDE.md` is owned by
[claude-sync](https://github.com/abmcar/claude-sync) (separate system) and
is NOT managed by DTVMDotfiles. The user-level file `@~/`-imports
`CLAUDE.shared.md` + `CLAUDE.local.md` from claude-sync. Both chains load at
session start; project overrides user where they conflict.

### Worktree symlinks

Git worktrees only contain tracked files. Dotfiles-managed content is
gitignored, so worktrees lack all config until `worktree-sync.sh` (or
`worktree-init.sh`) runs. The sync creates symlinks from worktree back to
the main repo — editing a symlinked path in the worktree writes to the main
repo. Run from the main repo:

```bash
bash DTVMDotfiles/worktree-sync.sh <worktree-path>
```

`CLAUDE.local.md` is per-machine and is NOT symlinked into worktrees.

### release.sh side effects

- Generates `.git/info/exclude` from `dotfiles/exclude.map.sh`
- Creates `AGENTS.md` and `GEMINI.md` as copies of `CLAUDE.md`
- Syncs `.claude/commands` to `~/.codex/prompts`
- Cleans up files that were in the previous manifest but no longer in dotfiles

## Common operations

| Goal | Command |
|------|---------|
| Add a new file to sync | Edit `MIRRORED_ITEMS` in `lib/sync_common.sh`, then `release.sh` |
| Check drift | `bash DTVMDotfiles/diff.sh` |
| Bootstrap a fresh DTVM workspace | `bash DTVMDotfiles/setup_from_dotfiles.sh` |
| Symlink dotfiles into a new worktree | `bash DTVMDotfiles/worktree-sync.sh <path>` |

## Output Requirements

When answering a DTVMDotfiles question, name the authoritative script,
state which direction the change flows (`release.sh` push vs `store.sh`
collect), and list any follow-up command.
